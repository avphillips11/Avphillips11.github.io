The Problem

A gap the security stack couldn't see

AI coding assistants have become standard-issue on knowledge-worker endpoints. Tools like GitHub Copilot, Cursor, and Claude Code execute Bash commands, read sensitive files, call APIs, and chain tool calls, all running with the user's credentials. But the enterprise security stack was never designed to see any of it.

EDR sees the process tree. SIEM ingests log forwarding. CASB inspects browser TLS. None of them see the AI tool decision: the moment an AI assistant decides to pass an API token to a Bash command, read a file containing customer data, or call an unauthorized MCP server. This is a design-era gap, not a configuration gap.

I identified this gap from inside a managed detection and response environment, using our own approved AI tools. I decided to build the solution. The constraint: I am not a software engineer. I do not write Python. What I know is security, product thinking, and how to direct a team. So I built the team out of AI.

The Approach

AI as Workforce

Rather than using Claude Code as a general-purpose assistant, I designed a structured workforce of 13 specialized AI team members, each with a detailed role profile, defined scope, working rules, and a growing archive of their own prior work that they review before every task. This is not prompt engineering. It is organizational design applied to AI.

The Product

What We Built

SkyNOPE fuses two independent endpoint witnesses: the customer's existing EDR (kernel-level) and Claude Code's lifecycle hooks (user-space, MDM-delivered), into one session timeline through a correlation engine. When the witnesses disagree, that disagreement is the security signal.

"Two witnesses, one console, neither can lie alone."

SkyNOPE 1.0 started as a single-witness file watcher: a Python agent reading Claude Code's session logs, feeding a Flask console. It proved the detection logic worked. But a single witness in user space meant an attacker who could tamper with the log files could blind the system entirely.

SkyNOPE 2.0 replaced the single-witness model with a dual-witness architecture. My security engineer and researcher independently identified the same structural limitation, then disagreed on the solution (EDR-first vs. hooks-first). I ran a structured reconciliation: each read the other's work, steelmanned the opposing position, and wrote a revised recommendation. Result: convergence on a hybrid model. Most of the disagreement was definitional, not substantive.

Design Principles

Why the model works

I
Agents develop institutional memory

Every agent reads their own prior work folder before starting a new task. The security engineer's threat models reference earlier threat models. Patterns improve across sessions.

II
Structured disagreement over consensus-seeking

When two agents disagreed on architecture, I didn't pick a winner. I spawned both again with instructions to steelman the opposing position. The result was genuine convergence, not false agreement.

III
Honest scope as a design constraint

The buyer persona reviewed the pitch deck three times. Overclaims were removed. The honest-scope statement was praised in the final review as the most credible security pitch evaluated.

IV
The AI executes under judgment. It doesn't replace it.

I made the strategic decisions. The AI team produced everything else. The bottleneck in building a product is rarely the ability to write code. It's the ability to define what should be built and why it matters.
